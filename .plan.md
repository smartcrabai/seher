# pi SDKプロバイダーのAPIエラーに exponential backoff リトライを追加

## Requirements

- HTTP 429 / rate-limit → リトライ対象: change needed（現状 `RunError::Limit` で即エラー返却。cruise側でリトライするが sleep なし。seher側でバックオフすべき）
- HTTP 5xx (500/502/503/504) → リトライ対象: change needed（現状 `RunError::Other` で即エラー返却。リトライ判定すら行われない）
- HTTP 401/404 → デフォルトはリトライしない。設定ファイルで `retry_client_errors: true` を指定した時のみリトライ対象: change needed（Kimi 等の一部プロバイダーは一時的な障害でこれらを返すことがある。実際に 2026-06-23 09:18 頃の障害では 401/404 が返ったが数分後には復旧していた。ただし401は通常APIキー失効を意味するため、opt-in にする）
- HTTP 403/400 → リトライ不要: not needed（権限なし・不正リクエストは恒久的な失敗）
- `RunAgentOptions` を `Clone` にする: change needed（現状 Clone 未実装だがリトライループで再利用に必要。全フィールドが既に Clone 可能: `SeherTool` は `Clone` 実装済み tool.rs:28、`CancelToken` は `Clone` 実装済み cancel.rs:9、その他フィールドはすべて標準型）
- 設定ファイルでリトライの有無・パラメータを変更できる: change needed（`config.yaml` の root レベルおよび provider レベルで `retry:` ブロックを書けるようにする）

## Design / Approach

**実装層**: `seher-sdk/src/sdk/config.rs`, `seher-sdk/src/sdk/errors.rs`, `seher-sdk/src/sdk/dispatch.rs`, `seher-sdk/src/sdk/resolve.rs`。cruise 側は変更不要。

**設定ファイルの記述例**:
```yaml
# root: 全プロバイダーのデフォルト
retry:
  enabled: true
  maxAttempts: 5
  initialDelaySecs: 2
  maxDelaySecs: 60
  multiplier: 2.0

providers:
  kimi:
    retry:
      enabled: true
      retryClientErrors: true  # 401/404 もリトライ（Kimi の一時障害対策）
    models: ...

  other-provider:
    retry:
      enabled: false   # このプロバイダーだけ無効化
    models: ...
```

**解決優先順**: `provider.retry` > root `retry` > ハードコードデフォルト（enabled=true, max_attempts=5, initial=2s, max=60s, multiplier=2.0）

**`SkillsConfig` と同じ解決パターン** (config.rs:127 `Config::resolve_skills` を参照):
- `Config::resolve_retry(&self, entry: &ProviderEntry) -> RetryConfig` を追加
- `ResolvedAgent` に `retry: RetryConfig` フィールドを追加
- `resolve_agent` (resolve.rs:238) で `cfg.resolve_retry(entry)` を呼んでセット

**リトライ判定**:
- `resolved.retry.enabled == false` → リトライしない（1回のみ）
- `RunError::Limit` → 常にリトライ（rate limit は必ず一時的）
- `RunError::Other { message, .. }` → 以下のいずれかが true ならリトライ:
  - `is_transient_http_error(message)` — 429/5xx（常時リトライ対象）
  - `resolved.retry.retry_client_errors && is_client_error_retryable(message)` — 401/404（opt-in）
- `RunError::Timeout` → リトライしない（タイムアウトは設定問題）

**バックオフ**（デフォルト値）:
```
attempt=1: run → 失敗 → sleep(2s)
attempt=2: run → 失敗 → sleep(4s)
attempt=3: run → 失敗 → sleep(8s)
attempt=4: run → 失敗 → sleep(16s)
attempt=5: run → 失敗 → Err を返す
```
sleep には `std::thread::sleep` を使用（`fold_stream` も同期ブロッキングなので問題なし）。

**API設計**: `run_for_resolved` 自体にリトライを内蔵する（`resolved.retry` を参照）。別関数は不要。外部からリトライ通知を受け取りたい呼び出し元のため `RunAgentOptions` に `on_retry: Option<Arc<dyn Fn(u32, &str) + Send + Sync>>` を追加する。

## Implementation Steps

1. `errors.rs` に2つの関数を追加:
   - `is_transient_http_error(msg: &str) -> bool` — 常にリトライすべきエラー:
     - `msg.contains("HTTP 429")` — 429 が `Limit` に分類されない場合のフォールバック
     - `msg.contains("HTTP 500")`, `"HTTP 502"`, `"HTTP 503"`, `"HTTP 504"` — 一時的なサーバーエラー
   - `is_client_error_retryable(msg: &str) -> bool` — `retry_client_errors: true` の時だけリトライすべきエラー:
     - `msg.contains("HTTP 401")` — 一部プロバイダーが一時障害で返す（Kimi 等）
     - `msg.contains("HTTP 404")` — 一部プロバイダーが一時障害で返す（Kimi 等）
   - 単体テストを errors.rs の `#[cfg(test)]` ブロックに追加

2. `config.rs` を修正:
   a. `RetryConfig` を追加（全フィールドに `serde(default)` を付ける。`Default` は enabled=true, max_attempts=5, initial_delay_secs=2, max_delay_secs=60, multiplier=2.0）
   b. `ConfigRaw`, `ProviderEntryRaw`, `Config`, `ProviderEntry` にそれぞれ `retry: Option<RetryConfig>` を追加
   c. `Config::resolve_retry` を実装（`SkillsConfig::resolve_skills` の構造を踏襲: provider > root > default）
   d. `Config::From<ConfigRaw>` で `retry` フィールドをコピー
   e. `ResolvedAgent` に `retry: RetryConfig` を追加

3. `resolve.rs` を修正:
   - L238 の `ResolvedAgent` 構築時に `retry: cfg.resolve_retry(entry)` を追加

4. `dispatch.rs` を修正:
   a. `RunAgentOptions` に `#[derive(Clone)]` を追加
   b. `RunAgentOptions` に `on_retry: Option<Arc<dyn Fn(u32, &str) + Send + Sync>>` フィールドを追加（Default: None）
   c. `run_for_resolved` の実装をリトライ対応に変更:
      ```
      pub fn run_for_resolved(resolved, prompt, opts) -> Result<RunOutput, RunError> {
          run_with_retry_inner(
              |p, o| { let rx = stream_for_resolved(resolved, p, o); fold_stream(&rx) },
              prompt, opts, &resolved.retry, opts.on_retry.as_deref(),
              std::thread::sleep,
          )
      }
      ```
   d. `run_with_retry_inner` を `pub(crate)` ヘルパーとして実装（`sleep_fn: impl Fn(Duration)` を受け取り、テストでゼロスリープに差し替え可能）

5. `mod.rs` にエクスポートを追加:
   - `config` の use 行に `RetryConfig` を追加
   - `errors` の use 行に `is_transient_http_error`, `is_client_error_retryable` を追加

6. `schemas/settings.schema.json` を更新:
   - `$defs` に `retryConfig` オブジェクト定義を追加。フィールド名は YAML のスネークケースではなくスキーマ慣例のキャメルケース（`maxAttempts`, `initialDelaySecs`, `maxDelaySecs`, `retryClientErrors`）。各フィールドに `description` と `default` を明記
   - root `properties` と `$defs.providerEntry.properties` に `retry: { "$ref": "#/$defs/retryConfig" }` を追加
   - 既存の `skillsConfig` / `skills` フィールドの定義・配置を参考にして一貫した書き方にする（schema L25-34, L71-73）

7. `README.md` を更新:
   - `### Format` の YAML 例（README L392-L435）に `retry:` の使用例を追加。`skills:` と並べて root レベルの例を示し、プロバイダーレベルでは `kimi:` のコメント付き例（`retryClientErrors: true`）を追記
   - `### Provider fields` テーブル（README L439-L448）に `retry.enabled`, `retry.maxAttempts`, `retry.initialDelaySecs`, `retry.maxDelaySecs`, `retry.multiplier`, `retry.retryClientErrors` の行を追加。`retryClientErrors` の説明には「一部プロバイダーが一時障害で 401/404 を返す場合に有効化する opt-in フラグ」と明記

8. `cargo test -p seher-sdk` でテスト通過を確認

## Testing Notes

**`is_transient_http_error` のテスト** (errors.rs):
- `is_transient_http_error("Anthropic API error (HTTP 500): internal")` → true
- `is_transient_http_error("Anthropic API error (HTTP 502): bad gateway")` → true
- `is_transient_http_error("Anthropic API error (HTTP 503): unavailable")` → true
- `is_transient_http_error("Anthropic API error (HTTP 504): timeout")` → true
- `is_transient_http_error("Anthropic API error (HTTP 429): rate limited")` → true
- `is_transient_http_error("Anthropic API error (HTTP 401): auth_error")` → false（401 は `is_client_error_retryable` の管轄）
- `is_transient_http_error("Anthropic API error (HTTP 404): not found")` → false（404 は `is_client_error_retryable` の管轄）
- `is_transient_http_error("Anthropic API error (HTTP 403): forbidden")` → false
- `is_transient_http_error("Anthropic API error (HTTP 400): bad request")` → false
- `is_transient_http_error("connection refused")` → false
- `is_transient_http_error("Read 50029 bytes")` → false（誤検知防止）

**`is_client_error_retryable` のテスト** (errors.rs):
- `is_client_error_retryable("Anthropic API error (HTTP 401): auth_error")` → true
- `is_client_error_retryable("Anthropic API error (HTTP 404): not found")` → true
- `is_client_error_retryable("Anthropic API error (HTTP 403): forbidden")` → false
- `is_client_error_retryable("Anthropic API error (HTTP 500): internal")` → false（5xx は `is_transient_http_error` の管轄）

**`RetryConfig` 設定解決のテスト** (config.rs):
- provider に `retry` なし、root に `retry` なし → Default（enabled=true, max_attempts=5, retry_client_errors=false）
- root に `retry: { enabled: false }` → enabled=false
- root に `retry: { max_attempts: 3 }` + provider に `retry: { enabled: false }` → provider 優先で enabled=false、max_attempts は root の 3
- provider に `retry: { max_attempts: 2 }` だけ → enabled は root/default の true を継承
- provider に `retry: { retry_client_errors: true }` → retry_client_errors=true（401/404 がリトライ対象に）
- root に `retry: { retry_client_errors: true }` + provider に `retry: {}` → provider 優先で retry_client_errors=false（provider が空の RetryConfig で上書き）

**`run_with_retry_inner` の挙動テスト** (dispatch.rs):
`sleep_fn` に `|_| {}` を渡して即リターンにしてテストを高速化。

テストケース:
- 1回目で成功 → Ok、リトライなし
- 1回目 Limit 失敗 → 2回目で成功 → Ok（on_retry が1回呼ばれる）
- 1回目 Other(HTTP 401) で `retry_client_errors: true` → 2回目で成功 → Ok（今回の実障害ケース）
- 1回目 Other(HTTP 401) で `retry_client_errors: false`（デフォルト） → 即 Err（リトライしない）
- 1回目 Other(HTTP 404) で `retry_client_errors: true` → 2回目で成功 → Ok（今回の実障害ケース）
- 1回目 Other(HTTP 404) で `retry_client_errors: false`（デフォルト） → 即 Err（リトライしない）
- 1回目 Other+transient(HTTP 500) 失敗 → 2回目で成功 → Ok（`retry_client_errors` に依らず常時リトライ）
- 1回目 Other+non-transient(HTTP 403) 失敗 → 即 Err（リトライなし）
- `RetryConfig { enabled: false }` 時に transient エラー → 即 Err（リトライしない）
- max_attempts 回すべて失敗 → 最後のエラーを返す
- sleep 計算: attempt=1→2s, attempt=2→4s, attempt=3→8s, attempt=4→16s（max 60s でクランプ）

## Anti-patterns to avoid

- HTTP 403/400 をリトライしない（権限なし・不正リクエストは恒久エラー。`retry_client_errors` の対象に含めない）
- `is_transient_http_error` に 401/404 を入れない（`retry_client_errors` によって制御される別の判定関数 `is_client_error_retryable` に分離する）
- `is_transient_http_error` で `"5"` や `"50"` の前方一致にしない（誤検知が生じる: "Read 5029 bytes" 等）
- `tokio::time::sleep` を `run_for_resolved` 内で使わない（同期コンテキストで呼ばれる。`std::thread::sleep` を使う）
- `RetryConfig` の各フィールドを独立してオーバーライドするのではなく、`RetryConfig` 全体を `Option<RetryConfig>` として扱う（一部フィールドだけ provider で指定、残りを root から引き継ぐ、という細かいマージは実装複雑度に見合わない）
- YAML キー名は camelCase にする（`retryClientErrors`, `maxAttempts`, `initialDelaySecs`, `maxDelaySecs`）。`SkillsConfig.includeClaude`（config.rs の `rename = "includeClaude"`）が先例。Rust フィールドはスネークケースで、serde `rename` で YAML 側を camelCase にマッピングする
